## 1) 공모주 달력 (청약일+상장일+검색)

### 문제
- 네이버 금융 IPO 페이지는 **이미 상장한 종목을 목록에서 제외** → 당일 상장 종목(예: **져스텍**, 2026-06-29 상장)이 달력에 안 나옴.
- 달력에 청약일만 표시되고 상장일은 표시 안 됨.
- 공모주 검색 기능 없음.

### 해결 (`app/datasources/fintech.py`)
- 출처 **38커뮤니케이션**(38.co.kr):
  - 공모청약일정(`?o=k`) → 종목명·청약일(YYYY.MM.DD~MM.DD)·확정/희망공모가·경쟁률·주간사
  - 신규상장(`?o=nw`) → 기업명·신규상장일·현재가·공모가
  - 두 페이지를 **종목명으로 병합** → 청약일·상장일 모두 채움.
- **네이버는 폴백**(38 실패 시, 임박 청약만).
- 검색: `ipo_calendar(limit, q=...)` → q 있으면 38 여러 페이지(과거 종목까지) 탐색.
- API: `GET /api/fintech/ipo?limit=40&q=종목명` (`app/web/api.py`).

### 38.co.kr 접속 함정 2가지 (VPS에서 둘 다 필요)
1. **구형 TLS** → 기본 requests는 `dh key too small` 실패.
   → `_legacy_session()`: `ssl_context.set_ciphers("DEFAULT@SECLEVEL=1")` + `OP_LEGACY_SERVER_CONNECT`.
2. **VPS 로컬 DNS(systemd-resolved)가 38.co.kr만 해석 실패**(네이버는 정상).
   → 시스템 DNS 안 건드리고 `_doh_resolve()`가 공개 DoH `https://1.1.1.1/dns-query`(IP리터럴이라 시스템DNS 불필요)로 A레코드를 받아 `_pin_host()`(socket.getaddrinfo 임시 패치, host→ip만 강제, SNI/Host/인증서검증 유지)로 접속. **자기완결적**(VPS 재구축돼도 동작).

### 프론트 (`app/web/static/index.html`)
- 달력: 청약일=**금색** 점 / 상장일=**파란색** 점으로 색 구분 + 범례.
- 날짜 클릭 → 그날 **청약 중 + 상장** 종목 동시 표시(상장일 셀 파랑 강조).
- 검색창(엔터/버튼/✕전체) 추가, 검색 시 해당 종목 있는 달로 이동.
- 기본 표시 월 = **이번 달**.
- "청약 임박" 요약 카드: 과거 상장 종목 제외(오늘 이후 청약 시작분만).

### 검증 (VPS 라이브 OK)
- source=`38커뮤니케이션`, 져스텍 청약 `2026.06.18~06.19` / 상장 `2026.06.29` / 현재가 17,530.
- 검색 "스팩" 44건.

> 종목명 주의: 사용자가 "저스텍"이라 해도 정식명은 **져스텍**.

## 2) 더듬이 규칙기반 종합 판단 — 추세/과열 분리

### 문제 (사용자 보고)
- 종목이 **정배열(상승)인데 "🔴 적극 매도/과열 -3.0"** 으로 떠서 매수인 줄 알았는데 sell로 보임 = **반대로 느껴짐**.

### 원인
- `index.html`의 규칙기반 패널(deuBody)이 7개 지표를 **한 점수로 합산**하던 모델인데, 역추세(평균회귀) 비중이 큼:
  - 역추세: RSI(2)·스토캐스틱(1)·볼린저(1)·52주(1)·PBR(1) = **6점**
  - 추세: MACD(1)·정배열(1.5) = **2.5점**
- 그래서 많이 올라 **과열·고점**인 종목은 정배열이어도 합계가 음수 → 매도로 뒤집힘.
- **부호 버그 아님**(각 지표 방향은 옳음). 설계상 역추세 모델이었던 것.

### 해결 (사용자 선택 = "추세·과열 분리 표기")
두 축으로 분리:
- **추세축**: 이동평균(ma20 vs ma60 — 화면 정배열/역배열 라벨과 일치) + MACD.
  → `upTrend`/`downTrend`는 **ma20 vs ma60** 기준. **정배열이면 절대 "적극 매도" 안 나옴.**
- **과열축**(`heatS`): RSI·스토캐스틱·볼린저·52주·PBR. (+ 저점/저평가, − 과열/고점)

판단 매트릭스:

| | 과열(heatS≤-2) | 정상 | 과매도/저점(heatS≥2) |
|---|---|---|---|
| **정배열(상승)** | 🟡 추세 양호·단기 과열(추격 주의) | 🟢 매수 우호 | 🟢 적극 매수(상승추세 눌림) |
| **역배열(하락)** | 🔴 적극 매도/과열(강과열 heatS≤-4) | 🔴 매도 우호 | 🟡 반등 시도 가능(약추세) |
| **혼조** | 🟡 과열·관망 | 🟡 중립·관망 | 🟡 저점권·관망 |

- 판단 행 아래 근거 표시: `종합 ±s · 추세 상승(정배열) · 과열도 과열(±heatS)`.
  - **주의**: 추세 표시 단어는 `upTrend/downTrend`(=라벨) 기준으로 통일 — `trendS` 부호(MACD 포함 단기지표)로 쓰면 "정배열인데 추세 하락(-1)" 모순이 남(이미 수정 완료).

### 신호 색 = 의미 (사용자 안내 완료)
- 🟢 = **매수** / 🟡 = **대기(관망 — 사지도 팔지도 않음)** / 🔴 = **매도**

### ⚠️ 되돌리지 말 것
- 단일 합산 점수(`s>=3 매수 / s<=-3 적극매도`)로 회귀하면 정배열 상승주가 다시 매도로 뜸.
- **LLM 더듬이 2·3**(`app/analysis/deudeumi_ai.py`)은 별개(Claude가 판단) — 이 부호 문제와 무관, 손대지 않음.
